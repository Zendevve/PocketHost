# Phase 3 Plan: Setup & Configurations Integration

## Objective
Allow users to dynamically adjust the memory bounds (`-Xmx`) and target a specific world map name (`--world`) when starting the Minecraft server.

## Tasks

### <task>
<name>Update Server Store with Configuration State</name>
<read_first>
- src/store/serverStore.ts
</read_first>
<action>
Add `config` object to `ServerState` interface:
```typescript
config: {
  memoryLimit: number; // in MB
  activeWorld: string; // Folder name, e.g. "world"
};
```
Initialize this object in the `useServerStore` default state with:
`memoryLimit: 2048`, `activeWorld: 'world'`

Update `actions` block with a setter:
`updateConfig: (patch: Partial<ServerState['config']>) => void;`

Implementation in `create()`:
```typescript
updateConfig: (patch) => set((state) => ({ config: { ...state.config, ...patch } })),
```
</action>
<acceptance_criteria>
- State tracks `memoryLimit` and `activeWorld`.
- `updateConfig` successfully deep-merges config changes into the store.
</acceptance_criteria>
</task>

### <task>
<name>Patch Native Caller Parameters</name>
<read_first>
- src/services/serverManager.ts
- src/services/downloadService.ts
</read_first>
<action>
Modify `startServer` in `src/services/serverManager.ts` to retrieve the current configurations and send them into the Expo module correctly.

1. Fetch values from store:
```typescript
const { memoryLimit, activeWorld } = useServerStore.getState().config;
```

2. Construct the full paths:
We need `jarPath` (to `server.jar`) and `worldDir` (to the named folder).
In React Native layer:
```typescript
import * as FileSystem from 'expo-file-system';
const serverDir = FileSystem.documentDirectory + 'server/';
const jarPath = serverDir + 'server.jar';
const worldDir = serverDir + activeWorld + '/';
```

3. Call the Native Module properly:
Update `ServerProcessModule.startServer()`:
```typescript
ServerProcessModule.startServer(jarPath, memoryLimit, worldDir);
```
</action>
<acceptance_criteria>
- `startServer` passes `jarPath`, `memoryLimit`, and `worldDir` matching the Kotlin definitions.
</acceptance_criteria>
</task>

### <task>
<name>Expose Configuration Controls in Dashboard UI</name>
<read_first>
- src/app/index.tsx
</read_first>
<action>
Import React Native `TextInput` and `Slider` from `@react-native-community/slider` (if installed, if not, build a simple +/- memory stepper using pure React Native `TouchableOpacity`).
Let's build a pure React Native +/- stepper to avoid new dependency resolution safely.

1. Read `config` from the store:
`const { config, actions } = useServerStore(state => state);`

2. Implement a `SettingsPanel` block rendering conditionally or sitting above the "Start Server" button, containing:
- **World Name**: A React Native `<TextInput>` bound to `config.activeWorld` using `actions.updateConfig({ activeWorld: text })`.
- **Memory Limit**: Two buttons `[-]` and `[+]` that subtract/add `512` from `config.memoryLimit`, alongside a `<Text>` showing `{config.memoryLimit} MB`.
Limit memory minimum to 512, maximum maybe 4096 depending on what's safe. Use `Math.max(512, newMem)`.

3. Ensure these configuration elements are `<View pointerEvents={status === 'idle' ? 'auto' : 'none'}>` or purely disabled visually when the server is `starting` or `running` to prevent mutating them while active.
</action>
<acceptance_criteria>
- Basic GUI allows modifying `activeWorld` string.
- Basic GUI allows shifting `memoryLimit` integers smoothly.
- Settings are disabled if the server is actively running.
</acceptance_criteria>
</task>
